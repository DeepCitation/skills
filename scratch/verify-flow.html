/verify Pipeline Reference

Interactive reference for the DeepCitation verification workflow — flow charts, step costs, citation rules, and failure modes.

What is /verify

The /verify skill is a three-phase evidence pipeline: it reads a source document, writes a claim-linked report body with precise in-text citation markers, and uses the DeepCitation CLI to produce a verified HTML artifact where every citation is anchored to a line in the original evidence.

📌 Trigger condition
/verify anywhere in the user prompt — regardless of phrasing. Always executes the full pipeline.
🎯 Output artifact
A *-verified.html file in CWD with every citation matched, highlighted, and confidence-scored by the CLI.
⚙️ CLI dependency
deepcitation@latest via npx — a bundled binary with its own HTTP transport. Never install external packages to work around it.
🔒 Hard invariant
Only the CLI produces HTML. If verify fails, the deliverable does not exist — never hand-build a verified-looking HTML.

Three phases — per-iteration detail
1
Prepare evidence
Read the authoritative source into context via the CLI reader
ext-read 1–30s 2–3 calls
⏱ Duration
PDF: 1–5s · URL/Office: 5–30s · Multiple sources add in parallel so only the slowest counts
🔢 Tool calls
Bash ×1 (mkdir + prepare piped together) · Read ×1 (summary file into context) = 2 calls. +1 Bash if auth triggered.
📥 Input tokens
Minimal for the Bash call (~100 tok). The Read output is what matters — deepTextPages for a 20-page PDF is typically 8–25k tokens added to context.
📤 Output tokens
~50–200 tok (model acknowledgement of the prepare result). The costly direction is input — the evidence text flowing into context window.
Why this matters: Loading deepTextPages into context is the RE2-based matching mechanism — evidence text in context improves citation accuracy. prepare is the only way to read evidence; never use Glob/Grep/Read on the raw source.
2
Respond with citations — single topic
Write body inline: report prose + CITATION_DATA block → .deepcitation/draft-body.md
write gen-time 1 call
⏱ Duration
Model generation — typically 5–20s for a 500–1500 word report. Scales with output length, not input length.
🔢 Tool calls
Write ×1 (body.md) = 1 call. The model reasons inline before writing — no extra Read calls.
📥 Input tokens
Context carries over from Step 1: system prompt + deepTextPages + user question. For a 20-page PDF this is typically 12–30k input tokens at this point.
📤 Output tokens
Report prose (~500–1500 words) + CITATION_DATA JSON block. Estimate: 1.5–4k output tokens. The CITATION_DATA adds ~50–200 tokens depending on citation count.
2
Respond with citations — parallel (multi-topic)
Agent A + Agent B run simultaneously, each writing a section file. Then merge.
2× agent +merge 3 calls
⏱ Duration
Agents run in parallel so wall time ≈ slowest agent, typically 15–40s per agent. Merge adds ~1s. Total Step 2 wall time: 16–41s.
🔢 Tool calls
Agent ×2 (parallel dispatch) + Bash ×1 (merge) = 3 calls. Each agent internally uses Write ×1.
📥 Input tokens
2× the single-topic input — each agent receives the full deepTextPages copy in its prompt. For a 20-page PDF: 24–60k input tokens total across both agents.
📤 Output tokens
Each agent writes ~half the report: 1–2.5k tokens each. Merge produces no new tokens. Total output: 2–5k tokens.
Token cost warning: The 2× input cost is intentional — each agent needs the full evidence to reason accurately. This is the largest single token spend in the pipeline. Multi-topic parallel is faster in wall time but doubles input cost vs single-topic.
3
Verify via CLI
CLI matches anchors, scores confidence, produces HTML artifact, opens it
ext-write 1–30s 2 calls
⏱ Duration
CLI verify: 0.5–5s typical, up to 30s worst case. Open command: <1s. Total Step 3: 1–6s typical.
🔢 Tool calls
Bash ×1 (verify + open chained with ||) = 1–2 calls depending on whether verify and open are chained or separate.
📥 Input tokens
~100–300 tok (the Bash command itself). The CLI handles matching server-side — no evidence text needed in context here.
📤 Output tokens
~100–400 tok — model summary line: 12/14 verified · 2 partial → topic-verified.html. Cheapest step in the pipeline.
Post-verify: Open with explorer.exe (WSL), xdg-open (Linux), or open (macOS). Verify runs exactly once — the API handles partial matches; unmatched anchors are flagged in output.

Step 2 — Respond with Citations
🔀
Single vs parallel dispatch
When to write inline vs spawn two sub-agents
routing decision
Single topic
Write body directly to .deepcitation/<draft>-body.md inline. Run verify on it.
2+ distinct topics
Spawn Agent A (citations 1+) → section-a.md and Agent B (citations 100+) → section-b.md in parallel. Then merge.
When the Agent tool is available, you MUST use parallel dispatch for multi-topic questions — not sequential writing. The non-overlapping citation ID ranges (1+ vs 100+) let the merge command renumber without collisions.
What could be better: The threshold "2+ distinct sections" is fuzzy. A more explicit rule — e.g., "2+ top-level headings expected in the output" — would reduce model judgment calls. Also, 3+ sections could benefit from a third agent band (200+).
🔗
Merge + verify (parallel path)
Join section files, renumber citations, then produce HTML
ext-write ~5–40s total

The merge command renumbers citations across both section files and joins them into a single body. Verification happens in the same && chain — if merge exits non-zero, the chain aborts before verify runs.

Merge failure = diagnostic mode. Read both section files to find the broken CITATION_DATA block. Common causes: empty body, JSON wrapped in a markdown ` ```json ``` ` fence, missing n field, truncated JSON. Fix then re-run merge — do NOT re-dispatch agents unless the error can't be fixed inline.

Domain model

Every citation connects two documents. Domain A is the asserting document (your report). Domain B is the authoritative evidence. The CLI's search locates Domain B text and frames it in the keyholeViewport — terse anchors produce tighter framing.

Format 1 — verbatim

When sourceMatch reads naturally as claimText in prose. Bold the term directly.

The invoice totals **USD 4,350.00** [1] for services by **Acme Corp** [2].

claimText === sourceMatch (isVerbatim: true). Place [N] immediately after the bold term.

Format 2 — labelled link

When prose has its own voice, or the source term doesn't fit naturally.

[revenue grew](cite:1 '$4.2 million') over the prior year

claimText = Domain A prose. sourceMatch = verbatim Domain B text inside single quotes.

sourceMatch selection rules
🎯
Brevity rules — ≤4 words, ≤40 chars
Pick the tersest unique identifier from Domain B first, then decide format
Quantifier-drop
"six ground floor commercial units" → ground floor commercial (3w) — drop the count
Clause-truncation
"will automatically convert into shares" → automatically convert (2w) — cite the trigger verb
Priority-label only
"Senior to payments for Common Stock" → Senior to (2w) — the tier marker IS the claim
Definition — cite the term
"Safe Price means the price per share equal to…" → Safe Price (2w) — term, not definition body
Tax threshold
"you can generally deduct only 50% of…" → 50% (1w) — cite the number
Multi-value list
"Sections 304, 305, 306, 354…" → Section 304 (2w) — cite one item or a category label
Self-check gate: Pick sourceMatch first. If it's 5+ words, trim. Ctrl+F test: could a reader search this phrase and find it uniquely? If it takes 5+ words to be unique, pick the noun head.
⚠️
Traps — heading anchors & index pages
Common cases where a superficially correct anchor actually fails
Heading-only anchors: If the same phrase appears as a section heading AND inside the operative sentence, anchoring the heading will verify against the wrong location. Always anchor the operative clause, not the heading.
Index/appendix/TOC pages: Never set p to an index page. If a dollar amount or term appears in both the body and an A-Z index, always point at the body text page where the operative rule lives. Index entries produce garbage sourceContext.
Category labels vs operative phrases: "commercial units" is a topic label. "parking requirements" or "no right of access" are operative rules. Prefer operative phrases that prove the claim.
📋
CITATION_DATA block format
Fields, page ID format, line ranges, and wrapping rules
n
Citation ID — unique integer. Reuse the same n in prose for the same fact; only one entry in the CITATION_DATA block per n.
k (sourceMatch)
Verbatim Domain B text — must be a contiguous substring of sourceContext. Never a paraphrase. Never with ellipsis.
p (page ID)
Format: "pageNumber_arrayIndex" from the <page_number_N_index_I> tag. Example: "3_0" = page 3, first occurrence.
l (lines)
Array of line IDs including the sourceMatch line plus 1–2 adjacent lines. Example: [19, 20, 21]. Gives the sourceContext enough surrounding text to frame the highlight.
Do NOT wrap the JSON in a markdown fence. The <<<CITATION_DATA>>> / <<<END_CITATION_DATA>>> delimiters are the only wrappers the parser recognizes. A ` ```json ``` ` fence causes a silent empty parse and breaks merge.

Cloud sandbox constraints
☁️
Cowork / cloud sandbox rules
Constraints that apply when CLAUDE_CODE_REMOTE == "true"
  • The CLI is a single bundled binary — installing external packages (undici, node-fetch) cannot affect its network behavior. Never attempt this.
  • Do not modify HTTP_PROXY / HTTPS_PROXY on individual commands. The CLI auto-detects the sandbox proxy.
  • The CLI enforces a 90-second hard ceiling on every request. Do not extend via & backgrounding, sleep loops, or timeout 600 npx.
  • When a command times out, the failure is in the network layer, not the CLI logic. Do not simplify input to retry.
  • Recognize __DC_ERROR__ {"recoverable":false} on stderr → hard stop, no retry.
What could be better: The sandbox detection is a single env var check ($CLAUDE_CODE_REMOTE). A more robust signal — e.g., checking for the presence of the proxy hostname in HTTP_PROXY — would make the constraints self-documenting without requiring the model to remember the env var name.

Common traps where the model's natural instinct produces a bad anchor — and the correct fix.

Trap ✗ Fix ✓ Root cause
"Junior to payment of outstanding indebtedness" (6w) "outstanding indebtedness" (2w) Distinctive noun carries the claim. Quantifiers and prepositions are filler.
claimText = **Equity Financing**, k = "when the company raises capital" k = "Equity Financing" (Format 1) k must be verbatim Domain B text, not a paraphrase. Reader clicks bold term → sees sourceMatch highlighted — they must be the same.
"Tooth Numbers 3 9 14 19 24 30" (7w) "Tooth Numbers" (2w) Multi-value fields: cite one value or the list header, not the whole list. The list IS the sourceContext.
"earliest to occur...prior to" (ellipsis) "earliest to occur" (3w) sourceMatch must be a contiguous substring. Ellipsis means it spans two regions → two citations.
"first 5 years of employment" (5w) "5 years" (2w) Cite the threshold, drop the qualifying phrase. Prose carries: "within the first 5 years [3] of employment".
"30% of the adjustable taxable income" (6w) "adjustable taxable income" (3w) Cite the distinctive noun, not the full clause. Prose: "capped at 30% of adjustable taxable income [5]".

Pipeline failure modes
🔴
Hard stops — non-recoverable
Conditions where the pipeline must halt
  • __DC_ERROR__ {"recoverable":false} on stderr → surface verbatim, stop
  • Auth fails AND script is unavailable AND DC_NON_INTERACTIVE=1 AND user doesn't provide a key
  • Claims file was used as evidence and no primary source can be found
Never build a hand-crafted HTML report as a fallback. Returning a verified-looking HTML from your own knowledge misrepresents unverified text as verified. If verify cannot complete, the deliverable does not exist.
🟡
Recoverable failures
Conditions with defined recovery paths
  • Auth needed: output contains "action needed" → run auth flow (script → browser, or manual key), retry prepare
  • Update available: run npm install -g deepcitation@latest, retry command
  • Merge exits non-zero: read both section files, find malformed CITATION_DATA, fix inline or re-dispatch the broken agent with the stderr output as context
  • Unmatched anchor in verify output: check for typos, shorten k to a more distinctive term — but do not re-run verify

Complexity analysis — what could be better
🔴
The remark plugin is boilerplate every consumer must copy
~20 lines of unist-util-visit AST manipulation to replace [N] markers inline

Every React integration must reproduce the remarkCitationMarkers plugin verbatim. The plugin itself isn't complicated, but it requires knowledge of the unist AST, unist-util-visit, and React-Markdown's custom component injection. It is copy-paste infrastructure with no room for error.

Optimal path: Export remarkCitationMarkers directly from deepcitation/react — consumers import it in one line instead of copying 20. This also means any future AST fix is deployed once. The recipe in the doc could shrink to 3 lines: import the plugin, pass it to remarkPlugins, and render.
🟡
Two verify APIs with overlapping surface
Simple API (verify) vs explicit API (getAllCitations + verifyAttachment) — the distinction isn't immediately obvious

The simple API (deepcitation.verify({ llmOutput })) is documented after the explicit API in Section 2.3, even though it's labeled "recommended". Readers who land mid-document may implement the explicit path unnecessarily.

Simple API
1 call. Pass llmOutput directly. Internal parsing handled. Use for 90% of cases.
Explicit API
3 calls: parse → inspect/filter → verifyAttachment. Only needed when you must filter citations before verification.
Optimal path: Lead with the simple API. Put explicit API in a collapsible "Advanced: filtering citations" subsection so the happy path is immediately visible.
🟡
attachmentId 24-hour TTL buried in Appendix D
Production-critical constraint not surfaced near prepareAttachments

The fact that attachmentId values expire after 24 hours and must be cached is noted only in Appendix D under a "Production note" callout. A developer following the main flow wouldn't see this until they hit expiry errors in production.

Optimal path: Add a ⚠️ callout directly in Section 1.4 (Prepare Sources): "Cache attachmentId — valid for 24 hours. Re-upload to refresh."
🟢
Golden Rules + Common Mistakes table — well placed
The upfront rules prevent the most common mistakes before they happen

The "Golden Rules" section at the top — especially the Wrong/Correct table — is an effective progressive disclosure pattern. It front-loads the most commonly-violated rules (citations.length, raw llmOutput to users, custom type definitions) before the developer writes a single line of code.

What works: The binary Wrong/Correct format removes ambiguity. Developers scanning for "how do I check if there are citations?" find the answer in 2 seconds without reading prose. The Quick Reference recipes immediately below serve the same function for full patterns.
🔵
Streaming pattern requires implicit knowledge
Buffer-until-end behavior for CITATION_DATA is a non-obvious constraint

Section 3.3 explains that the <<<CITATION_DATA>>> block always arrives at the end of the stream — which is why you must buffer the full response before parsing citations. This is a fundamental constraint that shapes the entire streaming integration pattern.

During stream
extractVisibleText(partial) safely strips any partial CITATION_DATA appearing mid-buffer for live display
After stream ends
Parse full response with getAllCitationsFromLlmOutput(fullResponse), then verify, then re-render with CitationComponent
Optimal path: A single diagram showing "citation data is appended at end of stream" would make the two-phase pattern (live text display → post-stream verify) immediately clear. Currently requires reading 3.3 carefully to understand why buffering is necessary.

Type invariants — the most common bugs
CitationRecord is NOT an array
Record<string, Citation> — keyed by 16-char hash. Use Object.keys(citations).length, never citations.length.
Never define your own types
Import Citation, Verification, CitationRecord, VerificationRecord from deepcitation. Custom definitions diverge silently.
Always strip before display
extractVisibleText(llmOutput) before any user-facing render. Raw output contains <<<CITATION_DATA>>> blocks.
getCitationStatus() for status checks
Never compare v.status === "found" directly — the status value table has 9 variants. Use getCitationStatus(v).isVerified.
attachmentId TTL
Valid for 24 hours. Cache after prepareAttachments(). Re-upload to refresh.
citationKey bridge
The same 16-char hash indexes both CitationRecord and VerificationRecord. Use getCitationKey(citation) — never compute manually.
